## Motivation for AOT JIT code

JavaScript engines utilize JIT compilation to produce code at runtime.
This is a practical necessity. Many useful semantic properties of a
JavaScript program, namely its types, are only discovered at runtime.
Type information is precisely what compilers need to emit fast code.
As a result, the design of Mozilla's SpiderMonkey JavaScript engine
embraces lazy compilation JIT code through various mechanisms. [Inline
Caches](links.inline_caches) are the first, lowest cost JIT mechanism. As instrumentation
suggests certain regions are executed more frequently, the JIT may
tier up into higher optimizations, such as [Baseline Compilation]() or
[Ion](), under the presumption that the optimized execution will
amortize the compilation cost.

There are a few intertwined motivations which make an AOT system for
SpiderMonkey compelling. First is to notice that much of the JIT code
an engine emits is deterministic and/or predictably generated.
Bootstrapping such code means we could potentially attain faster
startup by reducing runtime compilation. Additionally, an AOT format
exposes opportunities for offline optimizations. While runtime compilation
is made stringently efficient, we have a surplus of both time and compute
when JIT code is provided AOT. 

To contextualize, V8 has integrated bootstrapping into its engine in
the form of [heap-snapshotting](links.v8_heap_snapshots) and [embedded builtins](links.v8_embedded_builtins), since at
least 2018. Heap snapshotting captures the initialized heap state to
skip redundant work at startup, while embedded builtins provide the JS
standard library at compile time. The goal is not to rebuild V8's
architecture line-by-line. There are various SM-isms,
such as the runtime-generated baseline interpreter, self-hosted
builtins, CacheIR, and more, which flex and constrain the design space
in unique ways. But overall, starting with AOT bootstrapping - some way
to capture JIT code in relocatable format and reuse it in subsequent
builds - is a good starting point.

This work is part of an ongoing project to develop a Jitless mode for
Firefox. With that in mind, this post will focus on how we
bootstrapped one particular region of the engine - the Baseline
interpreter. Subsequent parts of this series will cover extending the
AOT infrastructure to JS builtins, Inline Cache stubs, and other areas
of the engine.


## Why a Generated Interpreter?

You may be aware that high-performance interpreters are commonly
written in assembly. Some advantages of doing so include attaining
[fine-grained control flow](links.threaded_code), [manual code
layout](links.code_layout), and escaping the C calling convention. In
this respect, SpiderMonkey's [Baseline
Interpreter](links.baseline_interp) is no exception. What may surprise
you to hear, however, is that the baseline interpreter is _generated
at runtime_.

What does it mean for the interpreter to be _generated_? Mechanically
speaking, a linear buffer is allocated on the heap, has instructions
written into it, is memory-mapped as executable, then jumped into
through a trampoline. This is the defining characteristic of JIT code,
so to say the baseline interpreter is _generated_ simply means it is
JIT code, like any other in the engine.

The more important question is, _why would an interpreter be
generated?_ JIT compilers produce runtime-generated code out of
necessity, due to the undecidability of guest-program semantics. But
an interpreter has deterministic codegen, regardless of the execution
context, doesn't it? So why should it not be a regular compiled object
available at link time? As we will see, runtime generation is an
extremely natural, ergonomic way to produce code in SpiderMonkey.
Additionally, runtime generated code can freely couple with already
initialized engine components, a facet which both yields efficiency,
and presents interesting challenges for providing such code AOT.

## Bootstrapping JIT Code

Actually writing an interpreter in pure assembly is a grueling task.
Instead, developers use high level languages to emit assembly for
them. In SpiderMonkey, this interface is called the
[MacroAssembler](links.masm). The baseline interpreter is specified
entirely in the MASM DSL, written once and automatically lowered for
each supported platform. But the MacroAssembler itself is _only
available at runtime_, suggesting we have a chicken and egg problem:
The MASM is part of the same build artifact as the engine, so how
would we invoke the MASM to generate the baseline interpreter before
the MASM itself is available? Such dependencies in code generation can
generally be solved with [bootstrapping](links.bootstrapping).
However, if this were the only reason preventing us from providing the
baseline interpreter AOT, our solution would be clear. We could simply
generate the interpreter, dump it to a file, attach it as a build
input and rebuild. That sounds too nice. And in fact it is.

<BlogImage src="/images/jitless_part1/bootstrap.svg" alt="Two-pass
bootstrap build pipeline" darkMode="invert" />

Since the interpreter is generated relatively late in the startup
lifecycle, the clever engineers at Mozilla noticed that they could
bake in addresses of runtime objects directly into the interpreter
code, such as [opcode handlers](links.opcode_handlers),
[profiling](links.profiling) and debugging handlers, and, most
importantly, the addresses in the [dispatch
table](links.dispatch_table). Baking in runtime addresses is great for
performance. We save the pointer dereferences otherwise necessary to
access fields of our runtime objects, and the branch predictor
benefits from seeing stable target addresses rather than values loaded
through indirection. This is also the very reason for which
bootstrapping the interpreter is difficult.

## Relocatable JIT Code

[ASLR](links.aslr) guarantees that our virtual address space is
randomized each time our program is loaded. Our pre-compiled
interpreter blob would dereference garbage pointers on any subsequent
run if we bootstrapped directly. We need something akin to
_relocations_ in dynamic libraries, where the dynamic linker resolves
references to relocatable addresses by patching the [GOT](links.got),
at load time. We went through the tedious process of identifying all
the runtime pointers embedded into the baseline interpreter. The total
count lands at 350 patch entries, falling into the following broad
categories:

- Dispatch table (256 entries): one per bytecode opcode. Each points
  to a handler _within the blob itself_, so it depends on where the
  blob was loaded in memory.

- Runtime data pointers (~10 kinds): `JSContext*`, `JitRuntime*`,
  `Realm*`, `wellKnownSymbols`, GC write barrier state, and profiler
  flags. These resolve to heap objects that move on every launch.

- External code pointers: VM wrapper trampolines, debugger trap
  handlers, and a handful of direct C++ function pointers.


### Strategy #1: Pointer Patching

When a runtime pointer is emitted via a `masm` method, it must be
tracked. A natural choke point is through `masm.movePtr`, along with
its friends, `masm.loadPtr` and `masm.branchPtr`. For now, we
constructed some wrappers around these interfaces to dispatch AOT
emission and tracking.

```c++
void MacroAssembler::moveAOTReloc(AOTRelocKind kind, Register dest) {
  if (!isAOT()) {
    // Fallback to the existing codegen when AOT generation is off.
    movePtr(ResolveAOTRelocCompileTime(kind, runtime()), dest);
    return;
  }
  switch (aot().strategy()) {
    case AOTStrategy::PatchPointers:
      aot().emitPatchableMov(RuntimePatch(kind, 0), dest);
      return;
    case AOTStrategy::RegisterIndirect:
      // Avoid patching by loading from a reliable pointer to the
      // Zone, from where all pointers can be attained via extra
      // arithmetic and loads.
      emitAOTRelocViaZone(kind, dest);
      return;
  }
}
```

When `moveAOTReloc` is called with the patching strategy, a
`RuntimePatch` is created, a struct which records the current offset,
the kind of pointer to patch, and an additional payload if necessary.
An array of `RuntimePatch` structs are serialized to the bottom of our
AOT blob along with some metadata (mostly static offsets) required to
reconstruct the `BaselineInterpreter` class on the load side.

```c++
class RuntimePatch {
  public:
    // Each patch has a tag, instructing what patch to apply, and a
    // targetOffset, representing where to apply.
    ExternalRefKind kind;
    uint32_t targetOffset;
    
    // Some patches require payloads.
    union {
      uint32_t handlerOffset;
      VMFunctionId vmId;
      DebugTrapHandlerKind dbgKind;
      AOTCppFunctionId cppFnId;
    };

    explicit RuntimePatch(ExternalRefKind kind_, uint32_t targetOffset_) :
      kind(kind_), targetOffset(targetOffset_) {}
    
    void apply(const PatchContext& pc) const;
  private:
    RuntimePatch() = default;
    uintptr_t getValueToPatch(const PatchContext& pc) const;
};

```

structure showing code and patches" darkMode="invert" />


With the AOT code, metadata and patches, we have sufficient context to
bootstrap. In a second build, we disassemble the code portion of the
blob into a `.s`, which can be registered as a build input. At load
time, the metadata portion of the blob is deserialized, along with the
patches, which can be applied in the new heap and runtime context.

### Strategy #2: Indirection Through a Pinned Register

While pointer patching was functional, it had a fatal property.
Patching requires flipping page permissions from executable to
writable and back. Syscalls are expensive! In addition, writing to the
AOT code means the same `.text` pages, which could otherwise be shared
across JSRuntime instances, must be CoW'd (Copy on Write) for each
instance. This motivated a second strategy. Similar to V8's
[kRootRegister](). We pin a register pointing to an
`AOTIndirectionTable`, a GOT-inspired structure in `.data`. Relocation
sites in the code resolve pointers through an indirect load using the
pinned register as a base. This strategy keeps the AOT code immutable and
shareable. We can still use the same MASM choke points to transparently
record and emit these relocation when AOT mode is flipped on.

<BlogImage src="/images/figures/aot_indirection_table.svg" alt="AOT indirection table structure showing pinned register, runtime data pointers, and dispatch entries" darkMode="invert" />

We can see that loading the interpreter from AOT code is considerably faster
than generating it. Unfortunately, while these results look great at
first glance, one hundred micro-seconds is miniscule in the grand scheme.
This delta is gathered from instrumentation intentionally placed right
before and after our AOT loading will make an impact. That being said,
generalizing AOT to other areas of the engine, such as self-hosted builtins
or IC stubs may have a more tangible impact.

## An End to End Example

Let's trace the full lifecycle of a single runtime pointer through the
AOT system. The opcode handler for `Symbol` in the baseline interpreter
loads a well-known symbol such as `Symbol.iterator` or
`Symbol.toStringTag`:

```c++
// BaselineCodeGen.cpp
template <>
bool BaselineInterpreterCodeGen::emit_Symbol() {
  Register scratch1 = R0.scratchReg();
  Register scratch2 = R1.scratchReg();
  LoadUint8Operand(masm, scratch1);

  masm.movePtr(ImmPtr(&runtime->wellKnownSymbols()), scratch2);
  masm.loadPtr(BaseIndex(scratch2, scratch1, ScalePointer), scratch1);
  masm.tagValue(JSVAL_TYPE_SYMBOL, scratch1, R0);
  frame.push(R0);
  return true;
}
```

The runtime pointer is emitted by `movePtr` which loads the address of the
well-known symbols table, which changes every launch. In
a normal JIT build, this compiles to a `mov reg, imm64` with the
pointer baked in as an immediate. In AOT mode, the same call is
transparently redirected. `MacroAssembler::movePtr` is overridden to
intercept pointer emission:

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::movePtr(ImmPtr imm, Register dest) {
  if (isAOT()) {
    auto slot = aot().indirectionTable()->findSlot(uintptr_t(imm.value));
    if (slot) {
      emitAOTSlotLoad(*slot, dest);
      return;
    }
  }
  MacroAssemblerSpecific::movePtr(imm, dest);
}
```

When in AOT mode, every `ImmPtr` is checked against a reverse lookup
on the indirection table. If the value matches a known slot, such as
`WellKnownSymbols` in this case, the direct `mov imm64` is replaced
with an indirect load through the table. If it doesn't match (e.g. a
NaN-boxing tag constant), the original codegen path run. The
indirection table itself is a flat array of `uintptr_t` slots, owned
inline by `JitRuntime`. Each slot is enumerated by the `AOT_SLOTS`
macro:

```c++
// AOT.h
#define AOT_SLOTS(V)                    \
  V(JSRuntimePtr)                       \
  V(JSContextPtr)                       \
  V(InterruptBits)                      \
  V(JitActivation)                      \
  V(ContextRealm)                       \
  V(WellKnownSymbols)                   \
  V(JitRuntime)                         \
  V(LastBufferedCell)                    \
  V(ProfilerEnabled)                    \
  /* ... pre-barriers, C++ fns, etc. */ \
  V(VMWrapperBase)

class AOTIndirectionTable {
  uintptr_t slots_[uint32_t(AOTSlot::Count)];
 public:
  static constexpr uint32_t offsetOfSlot(AOTSlot slot) {
    return uint32_t(slot) * sizeof(uintptr_t);
  }
  // Reverse-lookup: find the slot for a given pointer value.
  // Linear scan over ~30 slots; used only during codegen.
  mozilla::Maybe<AOTSlot> findSlot(uintptr_t value) const;
};
```

For now, `emitAOTSlotLoad` resolves a slot in two loads. First it reads the
cached table base pointer from the `BaselineFrame`, then indexes into
the slot. Pinning a register is on the TODO list, and would allow us to
get from two loads per indirection, down to one.

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::emitAOTSlotLoad(AOTSlot slot, Register dest) {
  // Load 1: BaselineFrame -> table base pointer
  loadPtr(Address(FramePointer,
                  BaselineFrame::reverseOffsetOfAOTTableBase()), dest);
  // Load 2: table[slot] -> value
  int32_t slotOff = int32_t(AOTIndirectionTable::offsetOfSlot(slot));
  loadPtr(Address(dest, slotOff), dest);
}
```

Finally, at load time the table is populated with live pointers during
`JitRuntime::initialize` The code is position-independent, the `.text`
pages are immutable and shareable across processes, and the handler
source is untouched.
