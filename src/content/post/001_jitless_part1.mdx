> What happens when you serialize ephemeral JIT code and bootstrap it
into a second build? Better yet, why would you want to do so?

## What is AOT JIT code?

JIT compilation involves producing code at runtime. For
JavaScript engines, this is a practical necessity. A guest
program's types are _discovered at runtime_, and type information
is precisely what compilers need to emit fast code. A common
technique to accrue this runtime information is through
instrumentation. In addition to executing the program, the
interpreter is instrumented to also record counters, types, etc.
When instrumentation suggests that compiling some region of the
program into an optimized form may be profitable, the engine does
so! What to compile, and when, is the core decision boundary for
a JIT. If we understand JIT code as any machine code generated on
the fly, then Ahead of Time (AOT) JIT code refers to the process of
providing JIT code to the engine _at startup_.

> **Note:** AOT JIT code is distinct from AOT JavaScript, which is the
> much harder problem of compiling a dynamic language ahead of time in
> full generality. See [AOT JavaScript](https://dl.acm.org/doi/10.1145/3393673.3276950)
> for a formulation of that problem.

## Motivation for AOT JIT code

There are a few intertwined motivations that make proving JIT
code AOT compelling. First, we observe that a significant portion
of JIT code is deterministic or predictable in nature. Providing
such code at compile time could help attain faster startup,
preventing the need to continually recompile ephemeral JIT code
across runtimes. Secondly, an AOT format exposes opportunities
for optimization. Ordinary JIT codegen is heavily time and
compute constrained, limiting the set of feasible
transformations. With an AOT format, we have a surplus of both
time _and_ compute. Lastly, certain platforms, like iOS or
embedded devices, make runtime code generation difficult or
outright impossible. Providing JIT code AOT means we could
improve the performance of Spidermonkey on these platforms.
 
This post explores the design space for making JIT code
position-independent. We target one region of the engine to
start, the _Baseline Interpreter_, and bootstrap it into an AOT
format. In subsequent posts, we may extend the AOT system to
Inline Caches and self-hosted JS builtins. These two areas,
particularly the builtins, offer the most fertile ground for
optimization.

## Why a Generated Interpreter?

High-performance interpreters are commonly written in assembly.
Advantages of doing so include attaining fine-grained control flow,
[manual code layout]({links.code_layout}), escaping the C calling
convention and more. SpiderMonkey's [Baseline
Interpreter]({links.baseline_interp}) is no exception. What may surprise
you, however, is that the baseline interpreter is _generated at
runtime_.

What does it mean for the interpreter to be _generated_? Mechanically
speaking, a linear buffer is allocated on the heap, has instructions
written into it, is memory-mapped as executable, then jumped into
through a trampoline. Treating data as code is the defining
characteristic of JIT compilers. When we mentioned that the Baseline
interpreter is _generated_, we mean simply that it is JIT code, like
any other.

More importantly, _why would an interpreter be generated?_ JIT
compilers produce runtime-generated code out of necessity. But an
interpreter has deterministic codegen, regardless of the program
flowing through it. So why should it not be a regular compiled object
available at link time? As we will see, runtime generation is an
ergonomic way to produce code in SpiderMonkey. Additionally, by virtue
of its late arrival in the lifecycle of a JIT, runtime-generated code
can freely couple with already initialized engine components. While
this process can yield efficiency, it is precisely this coupling that creates
interesting challenges for providing such code AOT.

## Bootstrapping JIT Code

Writing an interpreter in pure assembly is a grueling task. Instead,
developers use high-level languages to emit assembly for them. In
SpiderMonkey, this interface is called the
[MacroAssembler]({links.masm}). The baseline interpreter is specified
entirely using the MASM, written once and automatically lowered for
each supported platform. But the MacroAssembler itself is _only
available at runtime_. Here we have a classic chicken-and-egg problem:
the MASM is part of the same build artifact as the engine, so how
would we invoke the MASM to generate the baseline interpreter before
it is available? This kind of dependency is generally solved with
[bootstrapping]({links.bootstrapping}). However, if this were the only
reason preventing us from providing the baseline interpreter AOT, our
solution would be clear. We could simply generate the interpreter,
dump it to a file, attach it as a build input and rebuild. That sounds
too nice. And in fact it is.

<BlogImage src="/images/jitless_part1/bootstrap.svg" alt="Two-pass
bootstrap build pipeline" darkMode="invert" />

Since the interpreter is generated relatively late in the startup
lifecycle, the Mozilla engineers noticed they could bake addresses of
runtime objects directly into the interpreter code, such as [opcode
handlers]({links.opcode_handlers}), [profiling]({links.profiling}) and
debugging handlers, and the addresses in the [dispatch
table]({links.dispatch_table}). This saves the pointer dereferences
otherwise necessary to access fields of runtime objects.
Unfortunately, it is also what makes bootstrapping the interpreter
difficult.

How have other engines addressed bootstrapping? V8 has long used
[heap-snapshotting](https://v8.dev/blog/speeding-up-v8-heap-snapshots) and [embedded
builtins]({links.v8_embedded_builtins}) to improve their startup
performance. Heap snapshotting captures the initialized heap state to
avoid allocating and initializing the same engine internals for every
isolate. Embedded builtins, a technique more analogous to the AOT
formulation described here, effectively provides the JS standard library
at compile time.

Every language implementation has it's idiosyncrasies. Compared
to V8, various SM-isms such as the runtime-generated baseline
interpreter, self-hosted builtins,
[CacheIR](https://2023.splashcon.org/details/mplr-2023-papers/4/CacheIR-The-Benefits-of-a-Structured-Representation-for-Inline-Caches),
and more, flex and constrain the design space in unique ways. But
overall, AOT bootstrapping — some way to capture JIT code in
relocatable format and reuse it in subsequent builds — is a
natural starting point.

## Relocatable JIT Code

With engine internals already initialized, we mentioned
previously that absolute pointers are freely embedded into the
code. Since [ASLR]({links.aslr}) randomizes the virtual address
layout each time a program is loaded, the interpreter code would
dereference garbage pointers if we reused it directly. Our JIT
code needs to outlive a single process. To reuse it across
processes requires something akin to _relocations_. The first
step in this process is identifying exactly which symbols need
relocations. Thus began the tedious process of tabulating all the
categories and frequencies of runtime pointers within the
baseline interpreter. All in all, 350 pointers were embedded:

- Runtime data pointers (~10 kinds): `JSContext*`, `JitRuntime*`,
  `Realm*`, `wellKnownSymbols` and more. These point to heap objects
  that move on every launch.

- External code pointers: JSClass pointers, VM wrapper pointers,
  debugger trap handlers, and a handful of direct C++ function
  pointers. A notable property of these pointers is that many exist
  at link time, or could be made to. In the future, partial
  symbolization of the JIT code could offload these indirections to
  the program linker.


### Strategy #1: Pointer Patching

When a runtime pointer is emitted via a `masm` method, it
must be tracked. A natural choke point is `masm.movePtr`, along with
its analogues, `masm.loadPtr` and `masm.branchPtr`. We constructed
wrappers around these interfaces to dispatch AOT emission and
tracking.

```c++
void MacroAssembler::moveAOTReloc(AOTRelocKind kind, Register dest) {
  if (!isAOT()) {
    // Fallback to the existing codegen when AOT generation is off.
    movePtr(ResolveAOTRelocCompileTime(kind, runtime()), dest);
    return;
  }
  switch (aot().strategy()) {
    case AOTStrategy::PatchPointers:
      aot().emitPatchableMov(RuntimePatch(kind, 0), dest);
      return;
    case AOTStrategy::RegisterIndirect:
      // Avoid patching by loading from the AOTIndirectionTable
      // via a pinned register.
      emitAOTRelocViaTable(kind, dest);
      return;
  }
}
```

When `moveAOTReloc` is called with the patching strategy, a
`RuntimePatch` is created, recording the current offset, the kind of
pointer to patch, and an additional payload if necessary. An array of
`RuntimePatch` structs is serialized to the bottom of the AOT blob
along with some metadata (mostly static offsets) required to
reconstruct the `BaselineInterpreter` class on the load side.

```c++
class RuntimePatch {
  public:
    // Each patch has a tag, instructing what patch to apply, and a
    // targetOffset, representing where to apply.
    ExternalRefKind kind;
    uint32_t targetOffset;
    
    // Some patches require payloads.
    union {
      uint32_t handlerOffset;
      VMFunctionId vmId;
      DebugTrapHandlerKind dbgKind;
      AOTCppFunctionId cppFnId;
    };

    explicit RuntimePatch(ExternalRefKind kind_, uint32_t targetOffset_) :
      kind(kind_), targetOffset(targetOffset_) {}
    
    void apply(const PatchContext& pc) const;
  private:
    RuntimePatch() = default;
    uintptr_t getValueToPatch(const PatchContext& pc) const;
};

```

<BlogImage src="/images/figures/aot_blob.svg" alt="AOT blob
structure showing code and patches" darkMode="invert" />


With the AOT code, metadata, and patches, we have sufficient context
to bootstrap. In a second build, we disassemble the code portion of
the blob into `AOT.S`, which can be registered as a build input. At
load time, the metadata portion of the blob is deserialized, along
with the patches, which can be applied in the new heap and runtime
context. Below is the `AOT.S` scaffolding.

```asm
.section .text.aot,"ax",@progbits
.balign 16
.global bl_aot_text_start
.global bl_aot_text_end

// JIT code can be written in here in .byte directives
// from a masm buffer. 
bl_aot_text_start:  
bl_aot_text_end:

.section .rodata
.balign 16
.global bl_aot_container_start
.global bl_aot_container_end

// Additional metadata needed to deserialize the AOT code
// can be organized here.
bl_aot_container_start:
bl_aot_container_end:
```

### Strategy #2: Indirection Through a Pinned Register

While pointer patching was functional, it had one critical drawback.
Patching requires flipping page permissions from executable to
writable and back. Syscalls are expensive! In addition, writing to the
AOT code means the same `.text` pages, which could otherwise be shared
across `JSRuntime` instances, must CoW (Copy on Write) for each
instance. This motivated a second strategy. Similar to V8's
`kRootRegister` introduced in [embedded-builtins](), we pin a register
pointing to an `AOTIndirectionTable`, a GOT-inspired structure in
`.data`. Relocation sites in the code resolve pointers through an
indirect load using the pinned register as a base. This strategy keeps
the AOT code immutable and shareable. We can still use the same MASM
choke points to transparently record and emit these relocations when
AOT mode is flipped on. This strategy is what we ended up going with.

## An End to End Example

With an AOT mechanism established, we can trace the full lifecycle of
a single runtime pointer, from its emission, to relocation, to load
time calculation. Below is the handler for the `Symbol` opcode in the
Baseline interpreter. Its function is to load a "well-known
symbol" such as `Symbol.iterator` or `Symbol.toStringTag`. The way it
finds the symbol is by peeking back into the engine itself. While the
[WellKnownSymbols](https://searchfox.org/firefox-main/rev/e959a4adbef21a366443993ab176f43461d38c28/js/src/vm/Runtime.h#151) storage has a static lifetime, it is still heap
allocated, and therefore, of an indeterminate address. Since we
embed this address, we must provide a relocation.

```c++
// BaselineCodeGen.cpp
template <>
bool BaselineInterpreterCodeGen::emit_Symbol() {
  Register scratch1 = R0.scratchReg();
  Register scratch2 = R1.scratchReg();
  LoadUint8Operand(masm, scratch1);

  // BLOG_NOTE: We changed `ImmPtr` to `RelocImmPtr` to explicitly direct
  // the masm to create a relocation for this symbol. That's the only
  // codegen change necessary.
  masm.movePtr(RelocImmPtr(&runtime->wellKnownSymbols()), scratch2);
  masm.loadPtr(BaseIndex(scratch2, scratch1, ScalePointer), scratch1);
  masm.tagValue(JSVAL_TYPE_SYMBOL, scratch1, R0);
  frame.push(R0);
  return true;
}
```

In a normal JIT build with AOT disabled, this still compiles to a `mov
reg, imm64` with the pointer baked in as an immediate. The `movePtr`
overload on a `RelocImmPtr` argument will transparently skip emitting
a relocation in such cases. The `MOZ_UNLIKELY` hints to a
branch-predictor that the AOT should not be predicted taken. When AOT
is enabled, and we intend to emit relocations, we don't necessarily
care about efficient codegen. This code path will run only at
dump time, i.e when we are creating the AOT code to bootstrap from.
We have a surplus of time here. When we load the interpreter AOT,
we will skip this code-generation entirely.

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::movePtr(ImmPtr imm, Register dest) {
  if (MOZ_UNLIKELY(isAOT())) {
    auto slot = aot().indirectionTable()->findSlot(uintptr_t(imm.value));
    if (!slot)
        MOZ_CRASH("Failed to reverse lookup pointer unknown pointer!");
    emitAOTSlotLoad(*slot, dest);
    return;
  }
  MacroAssemblerSpecific::movePtr(imm, dest);
}
```

The `AOTIndirectionTable` has a slot for every pointer we may want to
provide relocations for. The `RelocImmPtr` can be verified via reverse
lookup in the indirection table. If the value matches a known slot,
such as `WellKnownSymbols` in this case, indirection is emitted. If we
somehow encounter a pointer which doesn't match, we can crash at
dump-time. The indirection table itself is a flat array of `uintptr_t`
slots, owned inline by `JitRuntime`. Each slot is enumerated by the
`AOT_SLOTS` macro:

```c++
// AOT.h
#define AOT_SLOTS(V)                    \
  V(JSRuntimePtr)                       \
  V(JSContextPtr)                       \
  V(InterruptBits)                      \
  V(JitActivation)                      \
  V(ContextRealm)                       \
  V(WellKnownSymbols)                   \
  V(JitRuntime)                         \
  V(LastBufferedCell)                    \
  V(ProfilerEnabled)                    \
  /* ... pre-barriers, C++ fns, etc. */ \
  V(VMWrapperBase)

class AOTIndirectionTable {
  uintptr_t slots_[uint32_t(AOTSlot::Count)];
 public:
  static constexpr uint32_t offsetOfSlot(AOTSlot slot) {
    return uint32_t(slot) * sizeof(uintptr_t);
  }

  // Reverse-lookup: find the slot for a given pointer value.
  // Linear scan over ~30 slots; used only during codegen.
  mozilla::Maybe<AOTSlot> findSlot(uintptr_t value) const;
  AOTSlot findSlotOrCrash(uintptr_t value) const;
};
```

At the time of writing, the pinned register indirection is not yet
working. Instead of one indirect load replacing each runtime pointer,
we have `emitAOTSlotLoad` resolve in two loads. First, it reads the
cached table base pointer from the `BaselineFrame`, then indexes into
the slot. Register pinning should be working by the next post. 

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::emitAOTSlotLoad(AOTSlot slot, Register dest) {
  // Load 1: BaselineFrame -> table base pointer
  loadPtr(Address(FramePointer,
                  BaselineFrame::reverseOffsetOfAOTTableBase()), dest);
  // Load 2: table[slot] -> value
  int32_t slotOff = int32_t(AOTIndirectionTable::offsetOfSlot(slot));
  loadPtr(Address(dest, slotOff), dest);
}
```

Finally, at load time, the table is populated with live pointers
during `JitRuntime::initialize`. The code is position-independent, the
`.text` pages are immutable and shareable across processes, and the
handler source is untouched. We can now run Firefox with an
AOT-provided baseline interpreter!
