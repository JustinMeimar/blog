## Motivation for AOT JIT code

JIT compilation is the process of producing code at runtime. For
JavaScript engines, this is a practical necessity. A program's types
are only discovered at runtime, and type information is what compilers
need to emit fast code. Mozilla's SpiderMonkey engine lazily compiles
JIT code through several such mechanisms.


What to JIT compile, and when, is dictated by the expected performance
gain. The compilation cost should be amortized by more efficient
execution. However, a JavaScript program's type semantics are
undecidable, even at runtime. Accordingly, JIT compilers progressively
refine the program into a more optimized form throughout execution.
The simplest JIT mechanisms are [Inline
Caches](links.inline_caches), operation-local data structures which
lazily compute and dispatch to type-specialized fast-paths for
polymorphic JS operators, like plus or property access. As regions of
the program are executed more frequently, the JIT may _tier up_ into
higher tiers, such as [Baseline Compilation]() or [Ion]().

A few intertwined motivations make an AOT system for SpiderMonkey
compelling. First, much of the JIT code an engine emits is
deterministic or predictably generated. Providing such code at compile
time could attain faster startup. Additionally, an AOT format exposes
opportunities for offline optimization. JIT codegen must normally be
fast, but with AOT, we have a surplus of both time and compute.

For context, V8 has long used
[heap-snapshotting](links.v8_heap_snapshots) and, since around 2018,
[embedded builtins](links.v8_embedded_builtins). Heap
snapshotting captures the initialized heap state to avoid allocating
and initializing the same engine internals for every isolate. Embedded
builtins, a technique more analogous to the AOT formulation described
here, roughly provides the JS standard library at compile time.
Compared to V8, various SM-isms, such as the runtime-generated
baseline interpreter, self-hosted builtins, CacheIR, and more, flex
and constrain the design space in unique ways. But overall, AOT
bootstrapping - some way to capture JIT code in relocatable format and
reuse it in subsequent builds - is a natural starting point.

This work is part of my Masters research to develop an AOT mode for
Firefox. It is in the early stages, so this post will provide an
overview of how we bootstrapped one region of the engine - the
Baseline interpreter. Subsequent parts of this series will cover
extending the AOT infrastructure to JS builtins and Inline Cache
stubs.


## Why a Generated Interpreter?

High-performance interpreters are commonly written in assembly.
Advantages include [fine-grained control flow](links.threaded_code),
[manual code layout](links.code_layout), and escaping the C calling
convention. SpiderMonkey's [Baseline
Interpreter](links.baseline_interp) is no exception. What may surprise
you, however, is that the baseline interpreter is _generated at
runtime_.

What does it mean for the interpreter to be _generated_? Mechanically,
a linear buffer is allocated on the heap, has instructions written
into it, is memory-mapped as executable, then jumped into through a
trampoline. This is the defining characteristic of JIT code. The
baseline interpreter is _generated_ in the sense that it is JIT code,
like any other in the engine.

More importantly, _why would an interpreter be generated?_ JIT
compilers produce runtime-generated code out of necessity, due to the
undecidability of guest-program semantics. But an interpreter has
deterministic codegen, regardless of the program flowing through it.
So why should it not be a regular compiled object available at link
time? As we will see, runtime generation is an ergonomic way to
produce code in SpiderMonkey. Additionally, runtime-generated code can
freely couple with already initialized engine components, which yields
efficiency but presents interesting challenges for providing such code
AOT.

## Bootstrapping JIT Code

Writing an interpreter in pure assembly is a grueling task. Instead,
developers use high-level languages to emit assembly for them. In
SpiderMonkey, this interface is called the
[MacroAssembler](links.masm). The baseline interpreter is specified
entirely in the MASM DSL, written once and automatically lowered for
each supported platform. But the MacroAssembler itself is _only
available at runtime_, suggesting a chicken-and-egg problem: the MASM
is part of the same build artifact as the engine, so how would we
invoke the MASM to generate the baseline interpreter before it is
available? This kind of dependency is generally solved with
[bootstrapping](links.bootstrapping). However, if this were the only
reason preventing us from providing the baseline interpreter AOT, our
solution would be clear. We could simply generate the interpreter,
dump it to a file, attach it as a build input and rebuild. That sounds
too nice. And in fact it is.

<BlogImage src="/images/jitless_part1/bootstrap.svg" alt="Two-pass
bootstrap build pipeline" darkMode="invert" />

Since the interpreter is generated relatively late in the startup
lifecycle, engineers noticed they could bake addresses of runtime
objects directly into the interpreter code, such as [opcode
handlers](links.opcode_handlers), [profiling](links.profiling) and
debugging handlers, and, most importantly, the addresses in the
[dispatch table](links.dispatch_table). This saves the pointer
dereferences otherwise necessary to access fields of runtime objects,
and the branch predictor benefits from seeing stable target addresses
rather than values loaded through indirection. This is also what makes
bootstrapping the interpreter difficult.

## Relocatable JIT Code

[ASLR](links.aslr) randomizes the virtual address layout each time a
program is loaded. A pre-compiled interpreter blob would dereference
garbage pointers on subsequent runs if we bootstrapped directly. JIT
code is designed to be ephemeral, so reusing it across processes
requires something akin to _relocations_. We identified all the
runtime pointers embedded in the baseline interpreter, totaling 350
instances across the following categories:

- Runtime data pointers (~10 kinds): `JSContext*`, `JitRuntime*`,
  `Realm*`, `wellKnownSymbols` and more. These point to heap objects
  that move on every launch.

- External code pointers: JSClass pointers, VM wrapper pointers,
  debugger trap handlers, and a handful of direct C++ function
  pointers. A notable property of these pointers is that many exist
  at link time, or could be made to. In the future, partial
  symbolization of the JIT code could offload these indirections to
  the program linker.


### Strategy #1: Pointer Patching

When a runtime pointer is emitted via a `masm` method, it must be
tracked. A natural choke point is `masm.movePtr`, along with its
analogues, `masm.loadPtr` and `masm.branchPtr`. We constructed
wrappers around these interfaces to dispatch AOT emission and
tracking.

```c++
void MacroAssembler::moveAOTReloc(AOTRelocKind kind, Register dest) {
  if (!isAOT()) {
    // Fallback to the existing codegen when AOT generation is off.
    movePtr(ResolveAOTRelocCompileTime(kind, runtime()), dest);
    return;
  }
  switch (aot().strategy()) {
    case AOTStrategy::PatchPointers:
      aot().emitPatchableMov(RuntimePatch(kind, 0), dest);
      return;
    case AOTStrategy::RegisterIndirect:
      // Avoid patching by loading from the AOTIndirectionTable
      // via a pinned register.
      emitAOTRelocViaTable(kind, dest);
      return;
  }
}
```

When `moveAOTReloc` is called with the patching strategy, a
`RuntimePatch` is created, recording the current offset, the kind of
pointer to patch, and an additional payload if necessary. An array of
`RuntimePatch` structs is serialized to the bottom of the AOT blob
along with some metadata (mostly static offsets) required to
reconstruct the `BaselineInterpreter` class on the load side.

```c++
class RuntimePatch {
  public:
    // Each patch has a tag, instructing what patch to apply, and a
    // targetOffset, representing where to apply.
    ExternalRefKind kind;
    uint32_t targetOffset;
    
    // Some patches require payloads.
    union {
      uint32_t handlerOffset;
      VMFunctionId vmId;
      DebugTrapHandlerKind dbgKind;
      AOTCppFunctionId cppFnId;
    };

    explicit RuntimePatch(ExternalRefKind kind_, uint32_t targetOffset_) :
      kind(kind_), targetOffset(targetOffset_) {}
    
    void apply(const PatchContext& pc) const;
  private:
    RuntimePatch() = default;
    uintptr_t getValueToPatch(const PatchContext& pc) const;
};

```

structure showing code and patches" darkMode="invert" />


With the AOT code, metadata, and patches, we have sufficient context
to bootstrap. In a second build, we disassemble the code portion of
the blob into `AOT.S`, which can be registered as a build input. At
load time, the metadata portion of the blob is deserialized, along
with the patches, which can be applied in the new heap and runtime
context. Below is the `AOT.S` scaffolding.

```asm
.section .text.aot,"ax",@progbits
.balign 16
.global bl_aot_text_start
.global bl_aot_text_end

// JIT code can be written in here in .byte directives
// from a masm buffer. 
bl_aot_text_start:  
bl_aot_text_end:

.section .rodata
.balign 16
.global bl_aot_container_start
.global bl_aot_container_end

// Additional metadata needed to deserialize the AOT code
// can be organized here.
bl_aot_container_start:
bl_aot_container_end:
```

### Strategy #2: Indirection Through a Pinned Register

While pointer patching was functional, it had a critical drawback.
Patching requires flipping page permissions from executable to
writable and back. Syscalls are expensive! In addition, writing to the
AOT code means the same `.text` pages, which could otherwise be shared
across JSRuntime instances, must CoW (Copy on Write) for each
instance. This motivated a second strategy. Similar to V8's
[kRootRegister](), we pin a register pointing to an
`AOTIndirectionTable`, a GOT-inspired structure in `.data`. Relocation
sites in the code resolve pointers through an indirect load using the
pinned register as a base. This strategy keeps the AOT code immutable
and shareable. We can still use the same MASM choke points to
transparently record and emit these relocations when AOT mode is
flipped on.

<BlogImage src="/images/figures/aot_indirection_table.svg" alt="AOT
indirection table structure showing pinned register, runtime data
pointers, and dispatch entries" darkMode="invert" />

Loading the interpreter from AOT code is considerably faster than
generating it. Unfortunately, while these results look great at first
glance, one hundred microseconds is minuscule in the grand scheme.
This delta is gathered from instrumentation placed directly around the
AOT load path. Generalizing AOT to other areas of the engine, such as
self-hosted builtins or IC stubs, may have a more tangible impact.

## An End to End Example

With the AOT mechanism established, we can trace the full lifecycle of
a single runtime pointer. Below is the opcode handler for `Symbol` in
the baseline interpreter. It loads a "well-known symbol" such as
`Symbol.iterator` or `Symbol.toStringTag` via a runtime pointer.

```c++
// BaselineCodeGen.cpp
template <>
bool BaselineInterpreterCodeGen::emit_Symbol() {
  Register scratch1 = R0.scratchReg();
  Register scratch2 = R1.scratchReg();
  LoadUint8Operand(masm, scratch1);

  masm.movePtr(ImmPtr(&runtime->wellKnownSymbols()), scratch2);
  masm.loadPtr(BaseIndex(scratch2, scratch1, ScalePointer), scratch1);
  masm.tagValue(JSVAL_TYPE_SYMBOL, scratch1, R0);
  frame.push(R0);
  return true;
}
```

The runtime pointer can be found in `movePtr`. In a normal JIT build,
this compiles to a `mov reg, imm64` with the pointer baked in as an
immediate. In AOT mode, the same call is transparently redirected to
emit a semantically equivalent routine through the `AOTIndirectionTable`.

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::movePtr(ImmPtr imm, Register dest) {
  if (isAOT()) {
    auto slot = aot().indirectionTable()->findSlot(uintptr_t(imm.value));
    if (!slot)
        MOZ_CRASH("Failed to reverse lookup pointer unknown pointer!");
    emitAOTSlotLoad(*slot, dest);
    return;
  }
  MacroAssemblerSpecific::movePtr(imm, dest);
}
```

In AOT mode, every `ImmPtr` is checked via reverse lookup in the
indirection table. If the value matches a known slot, such as
`WellKnownSymbols` in this case, the direct `mov imm64` is replaced
with an indirect load through the table. The indirection table itself
is a flat array of `uintptr_t` slots, owned inline by `JitRuntime`.
Each slot is enumerated by the `AOT_SLOTS` macro:

```c++
// AOT.h
#define AOT_SLOTS(V)                    \
  V(JSRuntimePtr)                       \
  V(JSContextPtr)                       \
  V(InterruptBits)                      \
  V(JitActivation)                      \
  V(ContextRealm)                       \
  V(WellKnownSymbols)                   \
  V(JitRuntime)                         \
  V(LastBufferedCell)                    \
  V(ProfilerEnabled)                    \
  /* ... pre-barriers, C++ fns, etc. */ \
  V(VMWrapperBase)

class AOTIndirectionTable {
  uintptr_t slots_[uint32_t(AOTSlot::Count)];
 public:
  static constexpr uint32_t offsetOfSlot(AOTSlot slot) {
    return uint32_t(slot) * sizeof(uintptr_t);
  }
  // Reverse-lookup: find the slot for a given pointer value.
  // Linear scan over ~30 slots; used only during codegen.
  mozilla::Maybe<AOTSlot> findSlot(uintptr_t value) const;
};
```

At the time of writing, the pinned register indirection is not yet
working. Instead of one indirect load replacing each runtime pointer,
we have `emitAOTSlotLoad` resolve in two loads. First, it reads the
cached table base pointer from the `BaselineFrame`, then indexes into
the slot. Register pinning should be working by the next post. 

```c++
// AOTMacroAssembler.cpp
void MacroAssembler::emitAOTSlotLoad(AOTSlot slot, Register dest) {
  // Load 1: BaselineFrame -> table base pointer
  loadPtr(Address(FramePointer,
                  BaselineFrame::reverseOffsetOfAOTTableBase()), dest);
  // Load 2: table[slot] -> value
  int32_t slotOff = int32_t(AOTIndirectionTable::offsetOfSlot(slot));
  loadPtr(Address(dest, slotOff), dest);
}
```

Finally, at load time, the table is populated with live pointers
during `JitRuntime::initialize`. The code is position-independent, the
`.text` pages are immutable and shareable across processes, and the
handler source is untouched. We can now run Firefox with an
AOT-provided baseline interpreter!
