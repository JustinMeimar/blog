Last time we picked on SpiderMonkey's [BaselineInterpreter](), the
second performance tier in the engine, because it was generated
determinsitcally at runtime. We mentioned how in practice, runtime
generation is actually a virtue of the internal code-generation
abstractions employed by SpiderMonkey, and that in all-in-all,
boot-strapping the interpreter to become AOT would not, by itself,
yield any meaningful performance.

What out AOT interpreter did provide however, was the ground on which
to elaborate a more-or-less transparent AOT mode for other JIT code in
the engine. Code which, unlike the baseline interpreter, could yield
startup improvements, observable code generation savings and improve
SM's current "Jitless" mode for security sensitive execution
environments.

What other JIT code are we talking about? Before answering that, let
us first classify JIT code into two categories, deterministic and
non-determinstically generated. The baseline-interpreter belongs to
the former category. Assuming standard engine flags and sufficient VM
execution to trip promotion to baseline, the same baseline interpreter
is generated each time, multiple times per process, sometimes an
unbounded number of times if running a [JS Worker Heavy program], in
which each worker receives a isolated JSRuntime. Determinstic JIT code
is an obvious target for AOT, but what about non-deterministic?

Non-deterministic JIT code is the idiomatic form, produced at runtime
based on the dynamically realized types of the guest-program flowing
through the engine. Arguably qualifying JIT code as such is vacuous,
and I should just be calling this "JIT code", but for the purposes of
this post it eludes to a meaningful property. Since guest programs can
create an unbounded quanity of unique types, the space of
type-specialized JIT code produced to run such code quickly is also
unbounded. As such, we refine our discussion of non-deterministic JIT
code for AOT to the question of _predictable_ or _statistically
likely_ JIT code given some circumstances.

At this point we can introduce the two initial categories of JIT code
we provide AOT in this post. First are Inline-Caches, the second are
JS builtins. If you are familiar with both of these already, feel free
to skip to the subsequent section. I will give a brief overview of
both.

Inline caches are small, operation local data-structures which are
lazily filled at runtime based on the observed types of the operands
flowing through the operation. In SpiderMonkey, IC's are a linked
list, with the tail of the linked list being a correctness
guaranteeing "fallback stub" to the generic VM routine. An IC being
type-specialized means it can avoid the cost of abstaction (?) paid by
the generic VM routine, and dispatch to fast native code. Some Pareto
frontier of IC stubs are a great candidate for AOT provisioning,
because, as is both intuitive and empirically demonstrated, a small
number of stubs assume the vast majority of cases in practice.

Second are JS built-ins. This is the JavaScript standard library
anyone familiar with the language will recognize. Think your
`Array.map`, `Array.filter`, etc. Builtins in SpiderMonkey have the
distinction.

An ahead-of-time (AOT) program has three (primary) distinct phases -
compile time, link time and runtime. At compile time, a single
compilation unit (usually a source file like `.cpp`, `.rs`) is
tokenized, parsed, analyzed and optimized by the compiler.At link time
(which can be invoked implicitly by the compiler), operates on _object

A JIT compiler involves a strange blend of these phases. It has the
ordinary Compile, Link and Run phases of any AOT program for the
engine itself, however, there runtime is a second order nesting doll,
itself comprised of another compile time and runtime inside itself.

This is a long winded way of saying, the notable characteristic of JIT

symbolized, contains embedded absolute addresses to various types of

what kind of C++ object `0x77ef...` points to).

JIT code simply allocates a buffer, writes machine code into it, maps
