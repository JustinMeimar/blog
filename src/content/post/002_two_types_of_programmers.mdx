> Because the internet definitely needs more opinions.

There two primary relationships a programmer holds towards to their
craft. The first is programming a means to an end. The second is
programming as an end in itself.

Those disposed to the former category of thought are more willing to
embrace AI tooling. They are more comfortable yielding control to the
machine, so long as the machine produces what they want. Sometimes
what these people want is _exclusively_ the end product. Whether or
not the code producing their end product is an entangled hair ball is
a matter of indifference. We call these people "vibe-coders."

The latter category, perhaps because machines have aggressively
encroached on their economic and social niche, tend towards a more
reactionary rejection of AI programming as a whole. I have a lot of
sympathy for this perspective. Unfortunately, [Programming as an
Identity](https://ratfactor.com/tech-nope2) has been dealt a crushing
blow. Engineering has always been about reduction to known practices.
"Known practices" is in the name. AI's are good at doing things which
are known, so long as you tell them.

Somewhere in between the ludite and vibe-coder is what people are
calling [_Agentic
Engineering_](https://x.com/karpathy/status/2019137879310836075?utm_source=the+new+stack&utm_medium=referral&utm_content=inline-mention&utm_campaign=tns+platform).
This term is not vibe-coder cope. It is a distinction, a middle
ground. The distinction is simple, Agentic Engineers recognize and
care about code quality, vibe-coders don't.

Code which is clear, obvious and uses the right data strucutres forms
[an already well recieved
philosophy](https://www.cs.unc.edu/~stotts/COMP590-059-f24/robsrules.html).
Unforuntately this is precisely the philosophy which makes idiomatic
code precisely in-distribution of language models. When writing
natural language, mode-collapse produces unstyalized, lifeless crap.
When writing software, mode-collapse is what we call "idiomatic". I
would concede that the idiomatic code is in fact not the mode - the
mode is verbose crappy boiler plate - but it _is in distribution_, and
can be easily invoked through a modicum of intention.

Job loss is a somewhat tangential discussion. I've seen [Jevons
Paradox](https://en.wikipedia.org/wiki/Jevons_paradox) being discussed
several times as the non-doomer case for programmer employment post
AI. What I can not forsee, however, is humans writing code by hand
entirely code by hand. Natural language is the final, highest level
abstraction of programming. It is the final achievement, and all
further improvements to code-producing abstractions will take place
within agentic frameworks, which impose some degree of discipline to
condition the generation. The human will always have the option to be
in the loop, but they may not need to be.

And of course, different types of software merit their own
"automatability score." I suspect anything at the application layer
will be thoroughly automated by the end of this year. The deployment
platforms are becoming _so frictionless and scalable it is astouning_.
Software lower in the software stack has a critical mid-section that
is different.

I use claude-code for fleshing out my research atop of Mozilla's
SpiderMonkey. If I went ahead with the first working version of the
AOT mode I developed, which was horrendously ugly, `#ifdef` mess which
bifurcated all of codegen, simply because it worked, there is no doubt
it would be classified as a failure case of AI slop. However, just
like writing software pre-AI, the discovery of the optimal
compression, representation and abstraction is an iterative process. I
prompt, scrutinize, compress, repeat until the thing is so simple that
if fits in my head. My role is now designing interfaces, nitpiking
patterns, identifying complexity, testing, and debugging. The laziness
that is associated with AI programming producing garbage code is a
correlated, but not causal consequence of using the tool. And I admit
to being lazy many a time, maintaining discipline is difficult while
opening Pandora's box. But it's something I'm learning to improve.

Everyone has an opinion on the internet these days. A common criticism
I see circulating is: "if AI is so productive, where are all the cool
things you have built?". This is a valid criticism. Billions of tokens
flow through models each day, but most AI enthusiasts show up empty
handed for all their evangelism. I attribute this not to AI's being
poor tools for code generation, but that _most ideas suck, mine
included_. There is also a very real, concerning phenomenon of
psychohancy creating [delusions of genius]().

That being said, words are cheap. Actions have cost. To put some skin
the game, and take the risk of looking like a fool (if I don't already
appear as one), I'm going to be developing in public. On March 17th, I
had the idea for [FrictionBox](https://frictionbox.app) while in the
shower. The vision is a micro-app platform of "Friction Gates", think
word puzzles, typing tests, breathing and memory exercises to solve
before opening social media. The business model will be like Obsidian,
developer friendly plugin ecosystem, free to use locally,
aesthetically pleasing, and with a small monthly tier for
cross-platform sync.

For a few hours every evening I developed it entirely with Claude Opus
4.6, running 2-3 agents in parallel on average. Within one week I have
a chrome extension published, Android app pending, 12 micro-apps in
the plugin catalog - an entirely viable MVP. I'm now squarely in the
position of *does my idea suck like the rest of them*, which, as far
as I'm concerned, is exactly where I should be. I've had many terrible
ideas before, I wouldn't be surprised if this is another, but I'm
going to give it a shot.
